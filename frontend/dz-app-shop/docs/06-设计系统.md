# 柏厚生活一期 - 全项目设计系统

**文档编号**：BHS-DS-20260310-V1  
**版本**：V1.0  
**编制日期**：2026-03-10  
**适用范围**：柏厚生活小程序、后台管理系统、运营页面、后续品牌数字化界面  
**品牌英文名**：`Baihou Life`

---

## 1. 文档目标

本文档用于建立柏厚生活一期全项目统一设计系统，作为小程序端、后台管理端、运营配置页面和后续扩展页面的视觉与交互基线。

本设计系统强调三件事：

- 统一品牌识别，避免前后台割裂
- 保持小程序端的生活方式表达和后台端的高效操作体验
- 通过组件、色彩、排版、间距和状态规则降低后续页面设计与研发成本

本文档为执行版设计基线。后续页面设计、前端实现、组件命名和视觉校验，均以本文档为准。

---

## 2. 设计原则

### 2.1 品牌气质

柏厚生活不是低价电商风格，也不是纯概念型艺术品牌。整体视觉应体现以下气质：

- 温暖克制：避免高饱和、大面积刺激色
- 空间感：界面应有留白和秩序，体现家居与场景感
- 专业可信：信息组织清晰，避免装饰过度
- 生活方式感：小程序端要能传达“空间、场景、审美、交付”

### 2.2 产品设计原则

- 品牌统一优先：同一品牌下，小程序与后台应共享核心颜色、圆角、阴影、状态语义
- 内容优先：商品图、空间图、场景图是核心内容，界面不应压过内容
- 操作清晰：后台重点保证信息密度、批量操作效率和状态可识别性
- 渐进增强：一期不追求复杂动效，重点保证可用性、性能和一致性
- 可扩展：设计 token、组件命名和状态颜色必须支持后续新增模块

## 3. 适用场景与端别策略

### 3.1 小程序端定位

小程序端是品牌展示与转化入口，视觉重点如下：

- 强化品牌氛围和空间感
- 强化商品图、场景图、Banner 图展示
- 降低操作负担，适配移动端拇指操作
- 强调登录、留资、预约、下单等关键转化动作

### 3.2 后台管理端定位

后台管理端是运营工具，视觉重点如下：

- 提高表格、表单、筛选、批量操作效率
- 保证状态、权限、异常反馈清晰
- 允许更高信息密度，但仍保留品牌统一感
- 在若依 Vue3 体系内做品牌化，而不是重造后台 UI 框架

### 3.3 统一与差异化规则

统一内容：

- 品牌主色与中性色体系
- 字体层级逻辑
- 圆角、边框、阴影、按钮状态语义
- 成功、警告、错误、禁用等系统状态色

差异化内容：

- 小程序端使用更强图片展示、更大留白、更低信息密度
- 后台端使用更高信息密度、更轻装饰、更强表格与表单可读性

## 4. 品牌视觉基线

### 4.1 品牌关键词

- 空间
- 材质
- 温度
- 生活方式
- 专业全案
- 城市服务

### 4.2 视觉方向

整体采用“暖中性现代家居”方向，不使用科技蓝、荧光色、游戏化渐变，也避免奢侈品式极端冷淡黑金风。

推荐视觉表达：

- 奶油白、暖灰、木质棕、墨灰作为基础
- 苔绿或橄榄绿作为辅助品牌色
- 重点 CTA 使用稳重暖色，不使用过亮橙红

## 6. 字体系统

### 6.1 字体策略

全项目优先使用平台原生可用字体，保持稳定加载和跨端一致性。

推荐字体栈：

- 中文：`PingFang SC`, `Hiragino Sans GB`, `Microsoft YaHei`, sans-serif
- 英文与数字：跟随系统无衬线字体，不额外引入复杂 WebFont

### 6.2 字号层级

#### 小程序端

```text
Display    28/36
H1         24/32
H2         20/28
H3         18/26
Body L     16/24
Body M     15/22
Body S     14/20
Caption    12/18
```

#### 后台端

```text
Page Title 24/32
Section    20/28
Card Title 16/24
Body       14/22
Form/Text  14/20
Table      13/20
Caption    12/18
```

### 6.3 字重规则

- 页面主标题：600
- 模块标题：600
- 正文：400
- 重点数值、价格、统计：600
- 超过两层信息时，不通过继续加粗来区分，优先通过字号、颜色、间距处理

## 7. 间距与栅格系统

### 7.1 间距基准

使用 4px 作为最小单位，建立统一 spacing token：

```text
4 / 8 / 12 / 16 / 20 / 24 / 32 / 40 / 48 / 64
```

### 7.2 小程序端布局

- 页面左右安全边距：16px
- 卡片内边距：16px
- 模块上下间距：24px
- 商品列表卡片间距：12px
- 吸底按钮区与内容区保持至少 24px 间隔

### 7.3 后台端布局

- 页面内容区左右内边距：24px
- 查询区与表格区间距：16px
- 表单项行间距：16px
- 弹窗内容区内边距：20px
- 卡片与卡片间距：16px 或 24px

### 7.4 圆角规范

```text
XS  6px
SM  8px
MD  12px
LG  16px
XL  20px
Pill 999px
```

使用建议：

- 输入框、标签、小按钮：8px
- 卡片、弹窗、筛选面板：12px
- Banner 卡片、大图模块：16px

## 8. 阴影与边框

### 8.1 阴影层级

```text
Shadow 1: 0 2px 8px rgba(47, 42, 38, 0.06)
Shadow 2: 0 6px 18px rgba(47, 42, 38, 0.08)
Shadow 3: 0 10px 28px rgba(47, 42, 38, 0.12)
```

### 8.2 使用规则

- 小程序卡片默认使用 `Shadow 1`
- 大 Banner、浮层、底部弹层可使用 `Shadow 2`
- 后台表格容器默认不加重阴影，优先用边框和浅底区分
- 同一页面不同时出现超过两种明显阴影层级

### 8.3 边框规则

- 默认边框色：`Border Default`
- 选中边框色：`Brand Primary`
- 错误边框色：`Error`
- 输入框聚焦时使用 1px 品牌色边框 + 轻投影，不使用高亮蓝色

## 9. 组件系统

### 9.1 按钮

按钮分为 5 类：

- Primary：主操作，如登录、提交、保存、确认
- Secondary：次主操作，如查看详情、继续编辑
- Tertiary：文字按钮或轻操作
- Danger：删除、关闭订单、作废等风险操作
- Ghost：弱边框操作，用于筛选和次级弹窗动作

按钮规则：

- 主按钮最少高度 44px
- 小程序主按钮宽度优先满宽或半宽，不做过多小号按钮
- 后台表格行内按钮优先文字按钮或小尺寸 ghost 按钮
- Loading 态必须禁用重复点击

### 9.2 输入框与表单

- 输入框高度统一 40px，后台搜索区可用 36px
- 表单标签靠上或靠左必须统一，单页不可混用
- 必填项用 `*` 标识，但仅在必要字段使用
- 错误信息显示在字段下方，不只用 toast

### 9.3 标签与状态 Tag

状态标签建议：

- 草稿：灰
- 上架/启用/成功：绿
- 待处理/待确认：黄
- 已关闭/已取消：中性灰
- 异常/失败：红

标签样式：

- 默认圆角 999px
- 高度 24px
- 文字 12px/18px

### 9.4 卡片

卡片是全项目核心容器组件。

小程序卡片：

- 用于 Banner、空间场景、商品推荐、设计师专区
- 强调图片、标题、副标题、价格、操作按钮

后台卡片：

- 用于统计概览、筛选区、详情信息块
- 强调标题、字段分组和视觉秩序

### 9.5 表格

后台表格规则：

- 表头底色使用极浅中性色，不使用深色表头
- 行高建议 48px
- 单元格优先显示一行，超长内容截断并提供 tooltip
- 操作列固定在右侧
- 批量操作区和筛选区视觉分离

### 9.6 弹窗与抽屉

- 简单编辑用弹窗
- 复杂表单和详情对比用抽屉
- 删除确认使用二次确认弹窗
- 移动端小程序优先底部弹层，不滥用中间弹窗

## 10. 小程序端专项规范

### 10.1 首页

- 首页应突出“空间场景家居领先品牌”
- 视觉结构建议：品牌头图 -> 区域选择 -> 核心品类 -> 空间场景推荐 -> 设计师专区 -> 线索/预约入口
- 首页首屏信息不超过 3 层视觉重点

### 10.2 商品列表

- 优先双列卡片布局
- 图片比例建议 4:5 或 1:1，保持统一
- 标题最多两行
- 价格区根据角色差异展示，不允许出现跳动布局
- 筛选栏固定吸顶，但高度要轻

### 10.3 商品详情

- 首屏应突出图片轮播、商品名、核心价格信息
- 参数、标签、区域和场景信息做分组展示
- 设计师价需要和指导价明确区分
- CTA 区固定底部：留资、预约、下单按优先级组合

### 10.4 我的页面

- 信息层级清晰，少装饰
- 订单、预约、线索、身份状态采用卡片分组
- 设计师身份与普通客户身份在视觉上应有明确差异

### 10.5 小程序交互要求

- 触控热区不低于 44x44px
- 所有关键请求有 loading 和失败重试
- 轮播、吸顶、下拉刷新等动效要克制
- 遵循微信小程序安全区和原生交互习惯

## 11. 后台管理端专项规范

### 11.1 页面结构

后台页面统一为：

- 页面标题区
- 查询筛选区
- 批量操作区
- 表格区
- 分页区

详情页统一为：

- 标题与状态区
- 基础信息卡片
- 业务信息卡片
- 操作记录或状态流转区

### 11.2 信息密度

- 一屏优先看到核心筛选和表格主体
- 不做过度留白，不走展示型营销页面风格
- 同类操作按钮集中排列，避免分散在多个角落

### 11.3 后台颜色使用

- 主品牌色只作为强调色，不用于整页大面积背景
- 表格页面以白底 + 浅灰区块为主
- 统计卡可使用 `Brand Primary Soft`、`Brand Accent Soft` 做轻区分

### 11.4 后台关键页面建议

- 商品管理：强调状态、区域、品类、上下架批量操作
- 品类管理：强调树结构和启停状态
- 区域管理：强调排序、启停和当前覆盖范围
- 线索管理：强调状态、分配人、跟进记录和导出
- 订单管理：强调状态机、支付状态、物流和异常处理

## 12. 图标与图片规范

### 12.1 图标

- 全项目统一使用线性或微圆角图标风格
- 不使用 emoji 代替功能图标
- 后台与小程序尽量复用同一图标语义

### 12.2 图片

- 商品图、场景图必须保持统一裁切策略
- Banner 图避免高密度文字堆叠
- 图片上出现文字时，需保证可读性和品牌一致性

### 12.3 占位图

- 未上传商品图、区域图、Banner 图时使用统一中性占位图
- 占位图风格要轻，不喧宾夺主

---

## 13. 动效规范

### 13.1 动效原则

- 动效服务于状态变化，不为“酷炫”而增加
- 优先使用透明度、位移、缩放，不使用复杂形变

### 13.2 建议时长

```text
Hover / Focus      150ms
Modal / Drawer     200ms
Tab / Filter       180ms
Page Section Reveal240ms
```

### 13.3 使用边界

- 小程序首页允许轻量进场动效
- 后台表格和表单禁止大量动效干扰操作
- 必须支持 `prefers-reduced-motion` 或等效降级策略

---

## 14. 响应式与适配

### 14.1 小程序

- 以主流手机宽度优先设计
- 兼容常见刘海屏和底部安全区
- 底部按钮区必须避开系统手势区

### 14.2 后台

- 设计基准宽度：1440
- 兼容最低宽度建议：1280
- 表格过宽时允许横向滚动，但必须固定关键列
- 复杂详情页在中等宽度下应自动分栏降级

---

## 15. 无障碍与可用性

- 正文文字最小 12px，核心内容不低于 14px
- 所有可点击元素必须有可见 hover 或 focus 状态
- 图标按钮必须有文字提示或 `aria-label`
- 表单错误信息必须可定位、可理解、可恢复
- 不依赖单一颜色表达状态，必要时增加图标或文本

## 17. 研发落地要求

### 17.1 小程序端

- 将颜色、圆角、间距抽成统一变量
- 页面开发不得随意新增“临时色值”
- 商品卡、Banner 卡、价格区、身份标签优先抽成复用组件

### 17.2 后台端

- 在 `RuoYi-Vue3` 中建立柏厚生活主题变量
- 菜单、表格、表单、标签、弹窗优先基于 Element Plus 主题能力和业务组件封装实现
- 不直接在单页里硬编码大量样式

### 17.3 设计与开发协作

- 新页面需先对照本设计系统选用现有组件和 token
- 若确需新增组件，应先判断是否可抽象为全局组件
- 超出本设计系统的视觉方案需先补充文档再实施

## 20. 结论

柏厚生活一期设计系统采用“一套品牌基线，双端差异表达”的策略：

- 小程序负责品牌呈现与业务转化
- 后台负责运营效率与信息管理
- 两端共享统一的颜色、排版、组件和状态语义

后续所有界面设计与前端开发，应先复用本设计系统，再做页面级实现。
