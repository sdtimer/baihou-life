## M1 身份认证与角色管理

### 1.1 功能说明

系统支持三类角色：普通客户（C端业主）、设计师（B端授权用户）、管理员（内部运营）。认证基于微信生态，采用微信一键登录 + 手机号授权方式，最大限度降低用户使用门槛。

### 1.2 角色体系

| 角色 | role 值 | 来源 | 能力边界 |
|:---|:---|:---|:---|
| 游客 | 0 | 未登录 | 浏览首页、产品列表（不含价格详情）、公开内容 |
| 普通客户 | 1 | 微信授权登录 | 查看指导价、提交留资/预约、订单操作 |
| 设计师 | 2 | 后台管理员手动创建 | 查看设计师专属折扣价、下载素材原文件、专属素材库 |
| 管理员 | 99 | 系统预置 | CMS 全部操作权限 |

**核心约束**：
- 设计师账号不开放自助注册，仅由管理员在后台创建并通过短信/微信邀请激活。
- 普通客户登录后无法自行升级为设计师。
- 权限变更（如停用设计师）需重新登录后生效。

### 1.4 实现逻辑

#### 前端职责

| 环节 | 实现要点 |
|:---|:---|
| 登录触发 | 用户主动点击登录按钮，调用 `wx.login()` 获取 `code` |
| 手机号获取 | 使用 `<button open-type="getPhoneNumber">` 组件，获取加密手机号数据 |
| Token 存储 | 登录成功后将 `token` 存入 `wx.setStorageSync('token')` |
| 请求拦截 | 全局请求拦截器自动在 Header 中附加 `Authorization: Bearer {token}` |
| 角色判断 | 全局 Store 维护 `userRole`，根据值控制 UI 元素的显隐 |
| Token 过期 | 拦截器捕获 401 响应，清除本地 Token，引导重新登录 |

#### 后端职责

| 环节 | 实现要点 |
|:---|:---|
| code 换取 openid | 调用微信 `code2Session` 接口，获取 `openid` 和 `session_key` |
| 手机号解密 | 使用 `session_key` 解密前端传来的加密手机号数据 |
| 用户匹配 | 优先以 `openid` 匹配，若为设计师则以手机号关联后台预创建的设计师记录 |
| JWT 签发 | Payload 包含 `uid`、`role`、`region_id`，有效期 7 天，支持刷新 |
| 权限中间件 | 每个受保护接口校验 Token 有效性 + role 是否满足接口最低权限要求 |

### 1.5 价格显示逻辑

根据用户角色动态计算展示价格，这是核心业务规则：

```
if (user.role == 0) {
    // 游客：不显示具体价格，显示"登录查看价格"
    display = "登录查看价格"
}
else if (user.role == 1) {
    // 普通客户：显示指导价
    display = product.guide_price
}
else if (user.role == 2) {
    // 设计师：同时显示指导价和设计师价
    designer_price = product.guide_price * product.designer_discount
    display = {
        guide_price: product.guide_price,        // 划线价
        designer_price: round(designer_price, 2)  // 专属价（红色高亮）
    }
}
```

**关键约束**：
- `designer_discount` 由后台按设计师分组配置，范围 0.5 ~ 1.0。
- 设计师价格计算在后端完成并返回，前端不持有折扣系数（防逆推）。
- 指导价为 0 或未设定时，前端显示"价格面议"。

## M2 区域化定位与展示

### 2.1 功能说明

柏厚生活覆盖多个城市（成都、佛山、武汉等），不同区域的可售商品、服务项目和供应商不同。区域选择是产品浏览的**前置条件**，影响全局数据范围。

### 2.2 区域数据模型

| 字段 | 类型 | 说明 |
|:---|:---|:---|
| `region_id` | String | 区域唯一标识（如 `chengdu`、`foshan`） |
| `region_name` | String | 展示名称（如"成都"、"佛山"） |
| `province` | String | 所属省份 |
| `center_lat` | Float | 中心纬度（用于距离匹配） |
| `center_lng` | Float | 中心经度 |
| `is_active` | Boolean | 是否启用 |
| `sort_order` | Int | 排序权重 |

### 2.3 区域确定流程

```
┌──────────────┐
│ 小程序启动      │
└──────┬───────┘
       │
       ▼
┌──────────────────┐     有     ┌──────────────┐
│ 本地是否缓存        │ ────────▶ │ 使用缓存区域    │
│ currentRegionId?  │           │ 正常加载内容    │
└──────┬───────────┘           └──────────────┘
       │ 无
       ▼
┌──────────────────┐     成功   ┌────────────────────┐
│ wx.getLocation()  │ ───────▶ │ 匹配最近的支持城市     │
│ 获取GPS定位         │          │ (Haversine距离计算)   │
└──────┬───────────┘          └──────┬─────────────┘
       │ 拒绝/失败                      │
       ▼                              ▼
┌──────────────────┐         ┌────────────────────┐
│ 弹出城市选择弹窗    │         │ 弹窗确认:             │
│ 手动选择城市        │         │ "您当前定位在{城市}，   │
│                    │         │  确认使用该区域？"      │
└──────┬───────────┘         └──────┬─────────────┘
       │                            │
       ▼                            ▼
┌──────────────────────────────────────┐
│ 设置 Store.currentRegionId            │
│ wx.setStorageSync('regionId', value)  │
│ 刷新当前页面数据                         │
└──────────────────────────────────────┘
```

### 2.4 实现逻辑

#### 前端职责

| 环节 | 实现要点 |
|:---|:---|
| 定位获取 | `wx.getLocation({ type: 'gcj02' })`，需在 `app.json` 声明 `requiredPrivateInfos` |
| 距离匹配 | 使用 Haversine 公式计算用户坐标与各区域中心点距离，取最近且 `is_active=true` 的区域 |
| 全局状态 | Vuex/Pinia Store 维护 `currentRegionId`，同时持久化到 `localStorage` |
| 切换入口 | 首页顶部展示当前区域名，点击可打开城市选择弹窗 |
| 数据联动 | `currentRegionId` 变化时，发布全局事件，各列表页监听并重新拉取数据 |

#### 后端职责

| 环节 | 实现要点 |
|:---|:---|
| 区域列表接口 | `GET /v1/config/regions`，返回所有已启用区域，按 `sort_order` 排序 |
| 全局过滤 | 所有产品查询接口**必须**接受 `region_id` 参数，后端据此过滤 `product.regions` 数组 |
| 缺省处理 | 若前端未传 `region_id`，后端返回 400 错误（不做默认区域降级，防止数据混乱） |

### 2.5 区域与商品的关联模型

采用"全国通配 + 区域覆盖"策略：

```
商品A.regions = ["ALL"]           → 全国所有区域可见
商品B.regions = ["chengdu"]       → 仅成都可见
商品C.regions = ["foshan","wuhan"] → 佛山和武汉可见
```

后端过滤 SQL 逻辑：
```sql
WHERE (regions @> '["ALL"]' OR regions @> '["foshan"]')
  AND status = 'on_shelf'
```

### 2.6 区域切换联动

区域切换后需重置的数据：

| 需重置项 | 处理方式 |
|:---|:---|
| 产品列表 | 清空列表，重新请求第一页 |
| 产品详情缓存 | 清空（不同区域可能同 SKU 不同价格策略） |
| 首页推荐内容 | 重新拉取（推荐位可按区域配置） |
| 购物车/订单 | 不清空（已产生的订单不受区域切换影响） |

### 3.5 三轴筛选实现

#### 查询逻辑

三个维度采用 **AND** 逻辑组合：

```
GET /v2/product/list?region_id=foshan
    &category_id=1
    &space=kitchen_bath
    &scene=minimalist
    &page=1&page_size=20
```

等价 SQL：
```sql
SELECT p.*
FROM products p
WHERE p.status = 'on_shelf'
  AND (p.regions @> '["ALL"]' OR p.regions @> '["foshan"]')
  AND p.category_id = 1
  AND p.space_tags @> '["kitchen_bath"]'
  AND p.scene_tags @> '["minimalist"]'
ORDER BY p.sort_order ASC, p.created_at DESC
LIMIT 20 OFFSET 0;
```

#### 前端交互

| 交互要素 | 说明 |
|:---|:---|
| Tab 切换 | 顶部 Tab 切换"按品类" / "按空间" / "按场景"三种浏览模式 |
| 标签栏 | 每种模式下展示对应维度的标签列表，支持单选 |
| 交叉筛选 | 选择某个空间后，可在右侧面板进一步选择品类/场景做交叉过滤 |
| 列表展示 | 瀑布流或网格布局，每项展示：主图、名称、价格（按角色） |
| 加载更多 | 滚动触底自动加载下一页，空列表展示"暂无该条件下的产品" |

#### 后端索引策略

为满足 <1s 首屏响应要求，需建立以下索引：

```sql
-- 品类+区域组合索引
CREATE INDEX idx_product_category_region ON products (category_id, status)
  WHERE status = 'on_shelf';

-- 空间标签 GIN 索引
CREATE INDEX idx_product_space_tags ON products USING GIN (space_tags);

-- 场景标签 GIN 索引
CREATE INDEX idx_product_scene_tags ON products USING GIN (scene_tags);

-- 区域 GIN 索引
CREATE INDEX idx_product_regions ON products USING GIN (regions);
```

### 3.6 品类树管理

品类支持后台灵活调整，运营规则：

- 新增品类：立即生效，前端自动展示。
- 停用品类：该品类下已上架商品自动变为"下架"状态。
- 调整层级：移动品类到另一个父节点，关联商品自动跟随。
- 最大层级：限定为 2 级（一级大类 + 二级品类）。

## M4 产品详情与媒体展示

### 4.1 功能说明

产品详情页是从浏览到转化的关键页面，需完整展示产品信息、规格参数、场景效果图和元素图，并根据用户角色差异化展示价格和操作按钮。

### 4.2 产品完整数据模型

| 字段 | 类型 | 说明 |
|:---|:---|:---|
| `id` | Long | 主键 |
| `name` | String | 商品名称（如"TOTO 浴室柜 800mm"） |
| `sku_code` | String | 唯一 SKU 编码 |
| `brand` | String | 品牌名称 |
| `model` | String | 型号 |
| `category_id` | Int | 关联品类 ID |
| `guide_price` | Decimal(10,2) | 指导价（元） |
| `price_unit` | String | 计价单位（"元/片"、"元/米"、"元/套"等） |
| `designer_discount` | Float | 设计师折扣率（0.50 ~ 1.00） |
| `regions` | JSON Array | 可售区域列表 |
| `space_tags` | JSON Array | 空间标签 |
| `scene_tags` | JSON Array | 场景标签 |
| `spec_params` | JSON Object | 规格参数（结构因品类而异） |
| `description` | Text | 产品描述 |
| `status` | Enum | 商品状态（draft/on_shelf/off_shelf/archived） |
| `sort_order` | Int | 排序权重 |
| `created_at` | DateTime | 创建时间 |
| `updated_at` | DateTime | 更新时间 |

**规格参数 `spec_params` 示例**：

```json

  "material": "陶瓷",
  "dimensions": "800×450×850mm",

```

### 4.3 媒体资源模型

| 字段 | 类型 | 说明 |
|:---|:---|:---|

### 5.2 安全下载机制

下载流程采用 Token 签名机制，防止非授权用户通过接口爬取原文件：

```
┌─────────────────┐
│ 设计师点击"下载素材" │
└──────┬──────────┘
       │
       ▼
┌─────────────────────────────────────┐
│ GET /v1/designer/asset-token         │
│ Headers: Authorization: Bearer {jwt} │
│ Params: { product_id, media_id }     │
└──────┬──────────────────────────────┘
       │
       ▼
┌─────────────────────────────────────┐
│ 后端校验:                              │
│ 1. JWT 有效且 role == 2               │
│ 2. 设计师账号处于"启用"状态               │
│ 3. media_id 对应文件 access_level      │
│    包含 designer                      │
└──────┬──────────────────────────────┘
       │ 校验通过
       ▼
┌─────────────────────────────────────┐
│ 生成带签名的临时下载 URL                  │
│ sign_url = cdn_url + "?token={sign}" │
│        + "&expires={timestamp}"       │
│ 有效期: 5分钟                           │
└──────┬──────────────────────────────┘
       │
       ▼
┌─────────────────────────────────────┐
│ 返回: { download_url, expires_at }    │
└──────┬──────────────────────────────┘
       │
       ▼
┌─────────────────────────────────────┐
│ 前端调用 wx.downloadFile(download_url) │
│ 下载完成后 wx.openDocument() 或         │
│ 提示"已保存到手机"                       │
└─────────────────────────────────────┘
```

### 8.3 微信支付流程

```
┌────────────┐
│ 用户确认下单  │
└─────┬──────┘
      │
      ▼
┌─────────────────────────────────┐
│ POST /v1/order/create            │
│ body: { items[], region_id }     │
│ → 创建订单，状态 = pending_pay    │
│ → 锁定商品价格（快照）              │
└─────┬───────────────────────────┘
      │
      ▼
┌─────────────────────────────────┐
│ POST /v1/payment/wechat/prepay   │
│ body: { order_no }               │
│ → 调用微信统一下单API              │
│ → 返回支付参数给前端                │
└─────┬───────────────────────────┘
      │
      ▼
┌─────────────────────────────────┐
│ 前端调用 wx.requestPayment()      │
│ 唤起微信支付界面                    │
└─────┬───────────────────────────┘
      │
      ├── 支付成功 ──▶ 前端跳转订单详情页
      │                  (轮询确认支付状态)
      │
      └── 支付取消/失败 ──▶ 保留待支付订单
                           (30分钟后未支付自动关闭)
```

**支付回调**：

```
┌─────────────────────────────────┐
│ 微信服务器回调                      │
│ POST /v1/payment/notify           │
│ → 验签                            │
│ → 更新订单状态为 paid               │
│ → 写入 transaction_id             │
│ → 返回 success 给微信              │
└─────────────────────────────────┘
```

### 8.4 实现要点

| 要点 | 说明 |
|:---|:---|
| 幂等性 | 支付回调可能重复推送，需以 `order_no` + `transaction_id` 做幂等判断 |
| 价格快照 | 下单时将商品价格快照到 OrderItem，后续价格变动不影响已有订单 |
| 超时关闭 | 定时任务每分钟扫描超过 30 分钟未支付的订单，自动关闭 |
| 安全验签 | 回调接口必须验证微信签名，防伪造请求 |
| 退款 | 一期暂不支持在线退款，走线下处理 |

## 附录B 全局错误码

| 错误码 | 含义 | 触发场景 |
|:---|:---|:---|
| 400 | 参数错误 | 缺少必填字段、格式错误 |
| 401 | 未认证 | Token 缺失或过期 |
| 403 | 权限不足 | 非设计师访问设计师资源、非管理员访问CMS |
| 404 | 资源不存在 | 商品/订单/预约不存在 |
| 409 | 状态冲突 | 重复提交留资、订单已关闭不可支付 |
| 422 | 业务校验失败 | 区域不支持、商品已下架 |
| 429 | 请求频率超限 | 同一用户短时间重复操作 |
| 500 | 服务器内部错误 | 未预期异常 |

统一响应格式：

```json
{
  "code": 200,
  "message": "success",
  "data": { ... },
  "timestamp": 1709366400000
}
```

错误响应：

```json
{
  "code": 403,
```
