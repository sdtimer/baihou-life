## 2. 建设目标与范围

### 2.1 建设目标

柏厚生活小程序用于承接一期品牌展示、区域化选品、留资转化、设计师赋能和订单履约入口，核心目标包括：

- 展示空间、场景、全案导向的品牌与商品内容
- 支撑按区域、品类、空间、场景进行产品筛选与浏览
- 支撑普通客户留资、预约、下单、支付
- 支撑设计师识别、折扣价展示和素材下载
- 与后台 CMS 配置、订单状态和区域配置保持一致

### 2.2 一期实施范围

一期小程序直接覆盖以下八个模块：

| 模块编号 | 模块名称 | 说明 |
|:---|:---|:---|
| M1 | 身份认证与角色管理 | 微信登录、Token、角色识别、价格权限切换 |
| M2 | 区域化定位与展示 | GPS 定位、城市选择、区域切换 |
| M3 | 选品中心（三轴筛选） | 品类/空间/场景组合筛选、搜索、排序、分页 |
| M4 | 产品详情与媒体展示 | 图文详情、规格参数、价格与素材权限展示 |
| M5 | 设计师素材分发 | 设计师素材下载、下载反馈、权限拦截 |
| M6 | 客户意向留资 | 留资表单、重复提交拦截、成功反馈 |
| M7 | 服务预约 | 预约创建、我的预约、状态追踪 |
| M8 | 订单与支付 | 下单、微信支付、订单列表与详情 |

### 2.3 不在本期范围

以下内容不纳入本期实施：

- H5 / APP 多端同步开发
- 积分、会员等级、营销裂变
- 直播、IM、社区内容
- 售后中心、退款、对账系统
- 多租户与渠道化改版

## 3. 技术基线与工程落位

### 3.1 基线选择

小程序端采用微信原生小程序技术栈作为正式实施路线，不保留 `uni-app` 双方案。

### 3.2 目录落位

按开发目录结构文档执行，小程序工程落位如下：

```text
baihou-life/
└── frontend/
    └── baihou-life-miniprogram/
```

### 3.3 工程命名约定

| 层级 | 命名 |
|:---|:---|
| 品牌展示名 | `Baihou Life` |
| 小程序工程目录 | `frontend/baihou-life-miniprogram` |
| 小程序接口前缀 | `/v1/`、`/v2/` |
| 鉴权 Header | `Authorization: Bearer <token>` |
| 区域 Header | `X-Region-Id` |
| 小程序角色值 | `0/1/2` |

### 3.4 页面与公共层落位规范

小程序代码结构固定如下：

```text
baihou-life-miniprogram/
├── src/pages/
│   ├── home/
│   ├── auth/
│   ├── product/list/
│   ├── product/detail/
│   ├── designer/assets/
│   ├── lead/form/
│   ├── appointment/create/
│   ├── appointment/list/
│   ├── order/confirm/
│   ├── order/list/
│   ├── order/detail/
│   └── profile/
├── src/components/
├── src/services/
├── src/store/
├── src/utils/
├── src/styles/
└── src/assets/
```

公共能力固定拆分为：

- `services/auth.js`
- `services/product.js`
- `services/order.js`
- `services/appointment.js`
- `services/lead.js`
- `utils/request.js`
- `utils/storage.js`
- `utils/location.js`
- `utils/auth.js`
- `store/user.js`
- `store/region.js`

### 3.5 小程序通用实现要求

- 所有请求统一通过 `utils/request.js` 封装，自动注入 Token 和 `X-Region-Id`
- 登录态、角色态、区域态都必须落缓存并提供全局读取能力
- 页面不能直接写分散请求逻辑，统一走 `services/*`
- 价格、按钮权限、素材下载入口必须由角色态统一控制，不允许页面各自写死分支

## 4. 模块实施顺序与依赖关系

为避免 UI 设计、前端开发和接口联调错位，一期小程序按以下顺序实施：

### 4.1 第一阶段：基础链路

- M1 身份认证与角色管理
- M2 区域化定位与展示

原因：

- 所有页面都依赖用户态或区域态
- 商品、预约、订单和素材都需要统一的身份与区域上下文

### 4.2 第二阶段：核心展示与转化

- M3 选品中心
- M4 产品详情与媒体展示
- M6 客户意向留资

原因：

- 商品列表与详情是转化主链路
- 留资与详情是一期最直接的转化入口

### 4.3 第三阶段：设计师与服务能力

- M5 设计师素材分发
- M7 服务预约

原因：

- 设计师功能依赖角色识别
- 预约依赖区域与用户基础信息

### 4.4 第四阶段：交易闭环

- M8 订单与支付
- 我的订单、状态追踪

原因：

- 订单依赖商品详情、身份、支付参数和后端状态机
- 交易闭环需在基础展示与服务链路稳定后实施

### 4.5 依赖矩阵

| 模块 | 依赖 |
|:---|:---|
| M3 选品中心 | M1、M2 |
| M4 产品详情 | M1、M2、M3 |
| M5 设计师素材分发 | M1，且 `role == 2` |
| M6 客户意向留资 | M1、M2、M4 |
| M7 服务预约 | M1、M2 |
| M8 订单与支付 | M1、M2、M4、支付回调与订单状态机 |

## 5. 模块级实施方案

### 5.1 M1 身份认证与角色管理

实施内容：

- 微信登录与 Token 获取
- 用户信息缓存
- 角色识别与页面权限切换
- 游客、客户、设计师三套价格和操作权限展示

前端要求：

- 接 `A01`
- 完成 `wx.login`、Token 存储、登录态刷新
- 当前阶段采用“测试态登录”口径：前端发起 `wx.login`，后端按测试映射返回账号与角色，不接真实 `code2session`
- 建立统一登录拦截与未登录态兜底
- `role=0/1/2` 在价格、素材按钮、下单入口处统一生效

### 5.2 M2 区域化定位与展示

实施内容：

- GPS 自动定位
- 拒绝定位时手动选城
- 当前区域缓存
- 区域切换联动页面刷新

前端要求：

- 接 `A02`
- 支持首进定位、二次进入读缓存
- 区域切换后清空商品列表缓存并重拉
- 所有依赖区域的请求必须带 `region_id`

### 5.3 M3 选品中心（三轴筛选）

实施内容：

- 品类树加载
- 品类 / 空间 / 场景三轴筛选
- 搜索、排序、分页滚动加载
- 空状态与筛选激活态

前端要求：

- 接 `A03`、`A05`
- 筛选逻辑严格按 AND 组合组织请求参数
- 支持 `keyword/category/space/scene/sort_by/pageNum/pageSize`
- 滚动分页为追加加载，不整页闪烁重绘

### 5.4 M4 产品详情与媒体展示

实施内容：

- 商品大图与媒体展示
- 规格参数、品牌、价格区
- 角色差异化价格
- 素材区与转化入口

前端要求：

- 接 `A04`
- 设计师用户显示 `designer_price` 与素材下载入口
- 普通客户仅显示指导价
- 游客显示“登录查看价格”
- 商品跨区域不可售时要有明确错误提示

### 5.5 M5 设计师素材分发

实施内容：

- 设计师素材分组展示
- 下载前换取临时 token URL
- 下载成功、失败、过期反馈

前端要求：

- 接 `A07`
- 先调用素材 token 接口，再执行下载
- 非设计师、已停用设计师、链接过期都要有独立提示
- 不允许普通用户在 UI 层看到下载主入口

### 5.6 M6 客户意向留资

实施内容：

- 留资表单
- 手机号校验
- 重复提交反馈
- 成功提交回执

前端要求：

- 接 `A06`
- 支持从商品详情带商品信息进入留资页
- 同一用户重复提交时明确提示，不能无反馈失败

### 5.7 M7 服务预约

实施内容：

- 预约创建页
- 我的预约列表
- 预约状态展示

前端要求：

- 接预约创建、列表、详情接口
- 表单需校验日期、手机号、服务类型
- `preferred_date` 不允许选择今天之前的日期
- 状态展示与后端状态机保持一致

### 5.8 M8 订单与支付

实施内容：

- 下单确认
- 测试态模拟支付调起
- 订单列表与详情
- 订单状态追踪

前端要求：

- 接订单创建、测试态预支付、订单列表、订单详情接口
- 当前阶段采用“测试态模拟支付”口径：前端先拿预支付参数，再走模拟支付适配层，不接真实商户支付
- 完成支付结果回跳、取消支付兜底、支付失败提示
- 状态映射严格使用：
  - `pending_pay`
  - `paid`
  - `processing`
  - `shipped`
  - `completed`
  - `closed`

## 6. UI/UX 页面设计实施规范

### 6.1 总体执行规则

- 每个页面都必须先出页面级 UI/UX 方案，再进入开发
- 页面设计统一使用 `ui-ux-pro-max` 方法论执行，输出页面结构、视觉方向、组件层级、状态设计和移动端交互细节
- 所有页面必须符合《06-设计系统》中的颜色、排版、间距、圆角、状态色和品牌气质要求
- 禁止直接套用通用电商模板、科技蓝后台风或高饱和促销风
- 小程序端整体气质必须保持“温暖克制、空间感、生活方式感”

### 6.2 统一视觉 token

小程序页面必须统一使用以下设计 token：

- 主色：`#6F5A45`
- 主色 Hover / 强调：`#5F4C3A`
- 辅色：`#7D8A63`
- 页面背景：`#F7F4EF`
- 卡片背景：`#FFFFFF`
- 次级背景：`#F2EEE8`
- 正文主色：`#2F2A26`
- 次级文字：`#6C655E`
- 边框色：`#DDD5CC`
- 页面安全边距：`16px`
- 卡片圆角：`12px`

### 6.3 页面状态覆盖要求

所有关键页面必须覆盖以下状态：

- 加载态
- 空状态
- 错误态
- 权限态 / 未登录态

所有表单页面必须覆盖：

- 默认态
- 聚焦态
- 校验失败态
- 提交成功态

所有列表页面必须覆盖：

- 首屏骨架
- 分页追加
- 无结果
- 筛选激活态

### 6.4 页面级设计要求

#### 首页

- 以品牌氛围、区域入口、Banner 和空间化商品推荐为核心
- 强调图片、留白和品牌感
- 不做强促销横幅和密集功能宫格

#### 商品列表页

- 强调三轴筛选效率、卡片层级和滚动体验
- 卡片信息以图片、名称、标签、价格为主
- 不做密集瀑布流和满屏营销标签

#### 商品详情页

- 强调大图、规格参数、价格层级和转化动作区
- 设计师价格和素材入口需清晰区分但不过度抢色
- 图文信息组织需体现空间感和材质感

#### 设计师素材页

- 强调身份识别、素材分组、下载反馈和权限边界
- 下载前后状态要明显，避免用户误判

#### 留资页与预约页

- 强调低负担表单和高完成率
- 字段数量控制在一期必要范围
- 重点突出主操作按钮和校验反馈

#### 我的订单 / 我的预约页

- 强调状态可读性、时间信息和下一步动作
- 状态标签颜色必须沿用设计系统语义色

#### 我的页面

- 强调身份、当前区域、订单/预约入口和设计师权益入口
- 不堆叠无效功能入口

### 6.5 适配要求

- 页面设计与实现必须优先适配常见微信小程序手机宽度
- 不允许只按单一机型视觉稿实现
- 长列表、长图和表单页要重点控制滚动体验与点击热区

## 7. 测试与验收标准

### 7.1 测试基线

测试执行以《04-测试设计》为准，小程序实施方案补充以下重点验收项：

- 登录成功后，普通客户显示指导价，设计师显示折扣价和下载入口
- 首次进入允许定位时自动匹配城市；拒绝定位时必须可手动切城
- 商品列表必须支持品类 / 空间 / 场景 AND 组合筛选
- 不传 `region_id` 查询商品时，接口返回 400，前端不得静默失败
- 商品详情在跨区域访问时正确提示不可售
- 普通客户访问素材下载接口返回 403，前端不显示下载按钮
- 同用户同商品 24 小时内重复留资被拒绝
- 预约禁止选择过去日期
- 微信支付成功后订单状态更新正确，取消支付和支付失败有明确回退
- 测试态模拟支付成功后订单状态更新正确，取消支付和支付失败有明确回退

### 7.2 UI/UX 验收要求

- 首页、列表、详情、表单、个人中心必须符合设计系统与页面级 UI/UX 方案
- 每个页面至少检查以下维度：
  - 视觉一致性
  - 点击热区
  - 滚动性能
  - 按钮层级
  - 空状态与错误反馈

### 7.3 真机测试要求

- 覆盖 iOS 和 Android
- 重点检查：
  - 图片加载
  - 滚动分页
  - 登录流程
  - 区域切换
  - 预约表单
- 测试态模拟支付
  - 素材下载
