## 1. 文档目的

本文档用于明确柏厚生活一期后台管理系统的开发计划、实施顺序、工程落位、模块范围、测试验收与上线准备要求。

本方案以现有项目文档为依据：

- 《00-开发目录结构》
- 《01-概要设计》
- 《02-数据库设计》
- 《03-接口与状态机设计》
- 《04-测试设计》

本方案不是汇报版摘要，而是研发执行版基线文档。后续后台管理系统的 clone、初始化、二次开发、联调、测试与上线，均以本文档为准。

## 2. 建设目标与范围

### 2.1 建设目标

柏厚生活后台管理系统用于支撑小程序一期运营与交付闭环，核心目标包括：

- 支撑商品、品类、区域、Banner 等基础运营配置
- 支撑设计师账号与折扣管理
- 支撑客户线索跟进与转化
- 支撑预约、订单的后台处理与状态流转
- 复用若依现有权限、菜单、日志、导出、定时任务能力，降低一期开发成本

### 2.2 一期实施范围

一期后台直接覆盖以下八个模块：

| 模块编号 | 模块名称 | 说明 |
|:---|:---|:---|
| B1 | 商品管理 | CRUD、上下架、批量操作、媒体资源管理 |
| B2 | 品类树管理 | 两级品类、排序、启停 |
| B3 | 区域管理 | 区域 CRUD、启停 |
| B4 | 设计师账号管理 | 创建、分组、折扣配置、启停 |
| B5 | 线索管理 | 列表、分配、状态流转、Excel 导出 |
| B6 | 首页推荐位管理 | Banner CRUD、定时上下线 |
| B7 | 预约管理 | 列表、状态流转、人员分配 |
| B8 | 订单管理 | 列表、详情、发货、关闭 |

### 2.3 不在本期范围

以下内容不纳入本期实施：

- 多租户能力
- 财务对账与结算系统
- 库存中心
- 售后服务工单
- 审批流引擎
- 独立 BI 报表平台

## 3. 技术基线与工程落位

### 3.1 基线选择

后台管理系统采用若依前后端分离 `RuoYi-Vue3` 作为二次开发底座。先 clone 官方工程，再在其基础上扩展柏厚生活业务模块。

### 3.2 目录落位

按开发目录结构文档执行，后台代码落位如下：

```text
baihou-life/
└── backend/
    └── ruoyi-baihou-life/
```

### 3.3 工程命名约定

| 层级 | 命名 |
|:---|:---|
| 品牌展示名 | `Baihou Life` |
| 后端工程目录 | `backend/ruoyi-baihou-life` |
| 若依业务模块 | `ruoyi-baihou` |
| Java 包名 | `com.ruoyi.baihou` |
| 数据表前缀 | `bh_` |
| CMS 路由前缀 | `/admin/` |

### 3.4 模块落位规范

业务代码统一落位于 `ruoyi-baihou`：

```text
ruoyi-baihou/
├── src/main/java/com/ruoyi/baihou/
│   ├── controller/admin/
│   ├── service/
│   ├── service/impl/
│   ├── mapper/
│   ├── domain/
│   ├── dto/
│   ├── vo/
│   ├── enums/
│   └── config/
└── src/main/resources/
    └── mapper/baihou/
```

### 3.5 复用若依能力

后台二开必须优先复用若依现有能力：

- 用户、角色、菜单、按钮权限
- 操作日志与审计
- Excel 导出
- 文件上传
- 定时任务 Quartz
- 分页组件与统一响应结构

不得为一期单独重造权限与后台框架。

## 4. 模块实施顺序与依赖关系

为避免前后端并发开发时依赖错位，一期后台按以下顺序实施：

### 4.1 第一阶段：基础主数据能力

- B2 品类树管理
- B3 区域管理

原因：

- 商品管理依赖品类与区域基础数据
- 小程序筛选与后台配置共享品类、区域主数据

### 4.2 第二阶段：核心商品运营能力

- B1 商品管理
- B6 首页推荐位管理

原因：

- 商品与 Banner 是首页和选品中心的直接运营入口
- 商品媒体、区域、标签是后续线索、预约、订单的业务基础

### 4.3 第三阶段：账号与转化能力

- B4 设计师账号管理
- B5 线索管理

原因：

- 设计师账号需和微信用户关联
- 线索管理需基于已存在的商品、区域、用户数据

### 4.4 第四阶段：履约闭环能力

- B7 预约管理
- B8 订单管理

原因：

- 预约、订单依赖前端提交流程、状态机和定时任务能力
- 履约模块业务复杂度最高，需放在基础运营能力稳定之后实施

### 4.5 依赖矩阵

| 模块 | 依赖 |
|:---|:---|
| B1 商品管理 | B2、B3 |
| B4 设计师账号管理 | 微信用户数据、设计师映射逻辑 |
| B5 线索管理 | B1、B3、用户数据 |
| B6 Banner 管理 | 区域数据、资源上传能力 |
| B7 预约管理 | 前台预约提交流程、状态机校验 |
| B8 订单管理 | 前台订单提交流程、支付回调、Quartz |

## 5. 模块级实施方案

### 5.1 B2 品类树管理

实施内容：

- 树形列表查询
- 新增、编辑、排序
- 启停控制
- 停用品类自动下架关联商品

后端要求：

- 提供树形结构返回
- 返回 `product_count`
- 停用时联动修改关联商品状态

前端要求：

- 提供树表展示
- 支持排序、状态切换、数量显示

### 5.2 B3 区域管理

实施内容：

- 区域列表
- 新增、编辑、启停、排序
- 区域缓存清理

后端要求：

- 支持 `region_id` 作为业务主键
- 启停后主动清理 Redis 区域缓存

前端要求：

- 支持区域状态筛选
- 支持排序与启停操作

### 5.3 B1 商品管理

实施内容：

- 商品列表、详情、新增、编辑、删除
- 批量上架、下架、归档
- 媒体资源维护
- 区域、空间、场景标签配置

后端要求：

- 接口与 `03-接口与状态机设计` 中 B01 保持一致
- 删除采用软删除，状态置为 `archived`
- 上架前校验必须通过以下前置条件：
  - `name`
  - `sku_code`
  - `category_id`
  - 至少一个 `region`
  - 至少一张图片

前端要求：

- 列表支持 `keyword/category/region/status` 筛选
- 支持批量操作
- 支持媒体资源和标签录入

### 5.4 B6 首页推荐位管理

实施内容：

- Banner 列表、创建、编辑、删除
- 配置跳转地址、展示区域、上下线时间

后端要求：

- 支持时间窗配置
- 支持区域化展示配置

前端要求：

- 提供时间范围选择
- 提供跳转配置和图片预览

### 5.5 B4 设计师账号管理

实施内容：

- 设计师列表
- 创建设计师
- 编辑、启停
- 分组管理
- 折扣配置

后端要求：

- 与 `bh_wx_user` 建立手机号映射关系
- 用户后续登录时可自动识别为设计师角色

前端要求：

- 支持分组管理
- 支持折扣和账号状态配置

### 5.6 B5 线索管理

实施内容：

- 列表查询
- 按区域、品类、状态、日期、是否分配筛选
- 分配、状态流转、跟进备注
- Excel 导出

后端要求：

- 复用若依 Excel 导出能力
- 支持 48 小时未分配提醒
- 手机号列表展示需脱敏

前端要求：

- 筛选项与导出条件保持一致
- 跟进和分配动作可直接在列表触发

### 5.7 B7 预约管理

实施内容：

- 预约列表
- 状态更新
- 人员分配

后端要求：

- 严格执行状态机规则：
  - `pending`
  - `confirmed`
  - `in_progress`
  - `completed`
  - `cancelled`
- 非法状态流转必须拒绝

前端要求：

- 页面按钮仅暴露合法状态操作
- 详情页显示完整预约状态链路

### 5.8 B8 订单管理

实施内容：

- 订单列表
- 订单详情
- 后台确认、关闭、发货
- 物流单号维护

后端要求：

- 严格执行订单状态机规则：
  - `pending_pay`
  - `paid`
  - `processing`
  - `shipped`
  - `completed`
  - `closed`
- 接入 Quartz：
  - 超时关闭订单
  - 自动确认收货

前端要求：

- 列表展示关键状态
- 发货时校验物流单号
- 详情页展示状态流转与关键信息

## 6. 接口、状态机与数据约束

### 6.1 接口约束

后台接口不得另起新规范，直接执行现有接口文档中的 CMS 章节：

- `/admin/products`
- `/admin/categories`
- `/admin/regions`
- `/admin/designers`
- `/admin/designer-groups`
- `/admin/leads`
- `/admin/banners`
- `/admin/appointments`
- `/admin/orders`

所有 CMS 接口鉴权等级统一为 `ADMIN`。

### 6.2 状态机约束

后台实现必须完全服从现有状态机设计，不得自行修改状态值。

关键状态枚举如下：

| 对象 | 状态值 |
|:---|:---|
| 商品 | `draft / on_shelf / off_shelf / archived` |
| 预约 | `pending / confirmed / in_progress / completed / cancelled` |
| 订单 | `pending_pay / paid / processing / shipped / completed / closed` |
| 线索 | `new / following / converted / abandoned` |

### 6.3 Service 层规则

所有状态流转统一通过 Service 层校验，禁止直接操作数据库改状态。

## 7. 实施排期

考虑到需要先 clone 若依基线再进行二次开发，一期后台建议按 6 周实施。

### 第 1 周：工程初始化与主数据基础能力

- clone `RuoYi-Vue3`
- 跑通原版前后端
- 新增 `ruoyi-baihou` 业务模块
- 初始化菜单、权限、基础表
- 完成 B2、B3

交付物：

- 可运行的若依前后端基础工程
- 品类和区域管理可用
- 菜单与权限配置完成

### 第 2 周：商品管理主线

- 完成 B1 后端接口
- 完成商品管理前端页面
- 完成媒体资源与上架校验

交付物：

- 商品 CRUD
- 商品状态流转
- 商品批量操作

### 第 3 周：Banner 与设计师管理

- 完成 B6
- 完成 B4
- 打通设计师与微信用户映射逻辑

交付物：

- Banner 管理页面
- 设计师账号与分组管理页面

### 第 4 周：线索管理

- 完成 B5
- 完成导出、分配、跟进逻辑
- 完成未分配提醒机制

交付物：

- 线索列表与导出功能
- 线索状态流转可用

### 第 5 周：预约与订单闭环

- 完成 B7
- 完成 B8
- 完成 Quartz 任务

交付物：

- 预约管理可用
- 订单管理可用
- 定时任务运行通过

### 第 6 周：联调、测试与上线准备

- 全量联调
- 缺陷修复
- 权限校验
- 预发部署
- 初始化数据导入

交付物：

- 测试报告
- 预发验证记录
- 上线准备清单

## 9. 测试与验收标准

后台验收直接绑定现有测试设计。

### 9.1 必测功能场景

- 商品 CRUD
- 商品上架前置校验
- 商品批量操作
- 品类停用联动
- 区域启停与缓存清理
- 设计师创建与关联
- 线索导出
- 后台越权访问校验

### 9.2 必测状态机场景

- 商品状态机合法/非法流转
- 预约状态机合法/非法流转
- 订单状态机合法/非法流转
- 线索状态机合法/非法流转

### 9.3 必测安全场景

- 普通用户访问 CMS 返回 403
- 手机号后台脱敏展示
- SQL 注入无效
- XSS 输出已转义

### 9.4 上线验收口径

以下条件全部满足，才可进入上线：

- 八个后台模块可正常访问
- 菜单、角色、按钮权限配置完成
- 状态机测试通过
- Quartz 定时任务验证通过
- 预发环境回归通过
- 初始化数据导入通过

## 10. 风险与应对

| 风险 | 说明 | 应对措施 |
|:---|:---|:---|
| 若依基线差异 | clone 版本与预期不一致 | 固定上游仓库与 tag/commit |
| 数据准备不足 | 商品图片、品类、Banner、区域资料不完整 | 第 1 周冻结模板并组织补数 |
| 状态机实现分散 | 开发时绕过 Service 直接改库 | 所有状态更新统一走 Service 层 |
| 履约联调滞后 | 订单、预约模块开发晚导致压缩测试时间 | 第 5 周前准备模拟数据与联调脚本 |
| 权限配置不一致 | 菜单、按钮、接口权限不匹配 | 每模块交付时同步输出权限清单 |
