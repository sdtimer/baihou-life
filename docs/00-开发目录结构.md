# 柏厚生活小程序一期 — 开发目录结构文档

**文档编号**：BHS-STRUCT-20260310-V1
**版本**：V1.0
**编制日期**：2026-03-10
**面向对象**：项目经理、前端工程师、后端工程师、测试工程师、运维工程师
**品牌英文名**：`Baihou Life`
**品牌目录名**：`baihou-life`

---

## 1. 文档目的

本文档用于统一柏厚生活项目在开发阶段的品牌英文命名、仓库目录结构、模块边界、资源归档方式与命名规范，确保前后端、测试、运维及后续扩展工作使用同一套目录语言。

核心原则如下：

- 品牌展示统一使用英文名 `Baihou Life`
- 代码目录、仓库名、资源目录统一使用短横线小写形式 `baihou-life`
- 数据库表、变量前缀、历史模块兼容保留已有 `bh` / `baihou` 体系
- 在不破坏既有设计文档和若依模块规划的前提下，建立后续可落地的标准目录结构

---

## 2. 品牌与命名约定

### 2.1 品牌标准名

| 类型 | 命名 |
|:---|:---|
| 中文品牌名 | 柏厚生活 |
| 英文品牌展示名 | `Baihou Life` |
| 英文目录名 | `baihou-life` |
| 英文仓库建议名 | `baihou-life-showcase` |
| 英文站点域名建议 | `baihou.life` |

### 2.2 命名使用规则

| 场景 | 规则 | 示例 |
|:---|:---|:---|
| 文档标题 / 对外展示 | 使用品牌展示名 | `Baihou Life` |
| 本地目录 / Git 仓库 / 前端工程 | 使用短横线小写 | `baihou-life` |
| Java 模块 / 包路径 / 服务标识 | 使用 `baihou` | `com.ruoyi.baihou` |
| 数据库表前缀 | 使用历史前缀 `bh_` | `bh_product` |
| 环境变量前缀 | 使用大写下划线 | `BAIHOU_LIFE_API_BASE` |

### 2.3 兼容性说明

现有概要设计中已约定若依新增业务模块为 `ruoyi-baihou`，该命名属于后端技术实现标识，不建议改为 `ruoyi-baihou-life`，原因如下：

- `ruoyi-baihou` 已与现有文档、包名、表前缀形成一致映射
- `baihou-life` 更适合仓库目录名和品牌资源层，不适合直接替换后端历史模块名
- 保持“品牌名一套、代码实现一套但可映射”的方式，迁移成本最低

建议统一映射关系如下：

| 层级 | 建议命名 |
|:---|:---|
| 品牌层 | `Baihou Life` |
| 项目根目录 | `baihou-life` |
| 若依业务模块 | `ruoyi-baihou` |
| Java 包名 | `com.ruoyi.baihou` |
| 数据表前缀 | `bh_` |

## 3. 项目根目录建议

如果后续建立正式开发仓库，建议根目录采用如下命名：

```text
baihou-life/
```

若当前工作目录仍沿用中文名，也建议在正式代码仓库、制品目录、部署目录中统一切换为英文目录名，避免以下问题：

- CI/CD 脚本处理中文路径时出现兼容性风险
- 跨平台构建时路径编码不一致
- 前后端部署路径不统一，影响自动化脚本维护

### 4.2 目录分层说明

| 目录 | 用途 | 命名原则 |
|:---|:---|:---|
| `docs/` | 项目设计、接口、测试、历史资料 | 中文文档名可保留，编号统一 |
| `frontend/` | 小程序前端代码 | 工程目录使用英文短横线 |
| `backend/` | 若依后端及业务模块 | 保留 `ruoyi-baihou` 历史实现名 |
| `database/` | 初始化、迁移、测试数据脚本 | 按用途拆分 |
| `ops/` | 部署配置、环境变量模板、运维脚本 | 仅存放环境与部署相关文件 |
| `assets/` | 品牌图、Banner、产品素材、设计师资料 | 不与代码工程混放 |

### 4.3 当前仓库落地约束

- 当前唯一有效的小程序主工程为 `frontend/baihou-life-miniprogram/`
- `frontend/dz-app-shop/` 视为历史遗留目录，不参与当前版本交付与测试
- 后续若执行仓库清理，应优先将 `frontend/dz-app-shop/` 迁出主仓或归档，而不是继续作为有效工程保留

## 5. 前端目录规范

### 5.1 小程序工程命名

前端工程建议命名为：

```text
frontend/baihou-life-miniprogram
```

### 5.2 页面级目录建议

```text
src/pages/
├── home/                # 首页
├── product/             # 商品列表/详情
├── filter/              # 筛选中心
├── appointment/         # 服务预约
├── order/               # 订单中心
├── designer/            # 设计师素材
├── profile/             # 我的
└── auth/                # 登录授权
```

### 5.3 前端命名规则

- 页面目录使用业务英文单词，禁止拼音与中文混用
- 组件目录统一使用复数名词，如 `components/`
- 接口按业务拆分，如 `services/product.ts`
- 品牌常量建议单独维护在 `src/constants/brand.ts`

建议品牌常量示例：

```ts
export const BRAND_NAME_ZH = "柏厚生活";
export const BRAND_NAME_EN = "Baihou Life";
export const BRAND_SLUG = "baihou-life";
```

## 9. 落地建议

为减少历史包袱与改名成本，建议分两步执行：

### 9.1 第一步：立即统一新增内容

- 新增目录、前端工程、脚本目录全部使用 `baihou-life`
- 文档中品牌英文名统一写为 `Baihou Life`
- 环境变量、部署目录、资源目录统一按本文档执行

### 9.2 第二步：兼容历史结构

- 后端业务模块继续使用 `ruoyi-baihou`
- 数据表前缀继续使用 `bh_`
- 现有设计文档中的 `BHS` 编号体系继续保留

该方案能够在不推翻当前设计稿的前提下，建立一套正式开发期可长期维护的目录标准。
